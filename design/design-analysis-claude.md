# Architecture Analysis: Parallel App

> Generated by Claude Code on 2026-03-13

## Overview

This is a **Next.js 15 issue tracker** (think Linear/Jira-lite) with a clean full-stack TypeScript architecture using a "GraphQL-as-BFF" pattern — a single `/api/graphql` route handles all data operations, keeping the API surface small and letting the client request exactly the shape of data it needs.

---

## Data Flow

```
Browser
  └── React Components
        └── URQL Client (attaches JWT in headers)
              └── POST /api/graphql
                    └── GraphQL Route (extracts user from JWT)
                          └── Resolvers (check context.user)
                                └── Drizzle ORM
                                      └── TursoDB (SQLite)
```

---

## Key Architectural Layers

### 1. Frontend — Next.js App Router with Route Groups

```
src/app/
  (auth)/        ← signin, signup — no auth required
  (dashboard)/   ← issues, projects, settings — JWT-protected
```

Route groups (`(auth)`, `(dashboard)`) are a Next.js convention — parentheses mean the folder name doesn't appear in the URL. Auth guards live in `(dashboard)/layout.tsx` via a `useEffect` that calls `redirect('/signin')` if no token exists.

### 2. GraphQL API — Custom, No Framework

The API at `src/app/api/graphql/route.ts` uses **raw `graphql-js`** + `@graphql-tools/schema` rather than Apollo Server or Pothos. The schema is built once at module load, and each request gets a context object containing the authenticated user:

```typescript
// route.ts — context is built per-request
contextValue: await context(req)  // { req, user: User | null }
```

### 3. Authentication — JWT + bcrypt, Stateless

- **Sign in/up** → bcrypt verifies password → JWT signed with a secret → returned to client
- **Per-request auth** → `getUserFromToken()` verifies the JWT and fetches the user from DB
- **Client storage** → JWT lives in `localStorage` (not httpOnly cookies), injected into every URQL request via `fetchOptions`

### 4. Database — Drizzle ORM + TursoDB

Drizzle is a **type-safe, SQL-first ORM** — it generates TypeScript types directly from your schema. TursoDB is a hosted SQLite service (libsql protocol). The schema defines two tables with a one-to-many relation:

```
users ──< issues   (one user has many issues)
```

### 5. Client-Side State — URQL + React Context

- **URQL** handles all server state (queries/mutations with a cache layer)
- **React Context** (`UserContext`) handles UI state: current user object + light/dark theme toggle

## Issue Status Lifecycle

```
BACKLOG → TODO → IN_PROGRESS → DONE
```

Defined as a TypeScript `enum` in `src/db/schema.ts` and mirrored in the GraphQL schema as `enum IssueStatus` — keeping the DB, API, and types in sync.

## Notable Design Decisions & Gaps

### Strengths
- **Type safety end-to-end**: Drizzle infers DB types, GraphQL schema is the API contract, TypeScript ties it together
- **Minimal dependencies**: No heavy ORM or GraphQL server framework — just the primitives
- **Collocated tests**: `*.test.tsx` files live next to their components

### Things to Watch
- **No middleware-level auth**: Protection relies on client-side redirect + resolver-level `context.user` checks. A hardened production app would add Next.js middleware JWT verification too.
- **Hardcoded JWT secret**: `"use_an_ENV_VAR"` in `src/utils/auth.ts` is a known placeholder — must be an environment variable before production.
- **localStorage for JWT**: Not protected against XSS. httpOnly cookies would be more secure for a production auth system.
